Challenges & Aim

Why the migration?

Red Tiger is an old CMS that lacked flexibility to create sites, pages and marketing content.
It did not have sufficient authoring capabilities bacasue of which business was restricted to displaying content and products in a particular manner.

Challenges
  • Website creators and marketing personnel need more flexibility to display information and products.
  • The tool needs to be scalable to accomodate more themes, designs, layouts and customizations.
  • Version updates, design updates or policy changes are not automated and require a lot of engineering effort.
  • Any upgrade to features or designs require a complete development cycle.
  • No in-built or out of the box components to use.
Aim
  • To move to a more widely used and agile content management tool.
  • Use Adobe's OOTB components and grid system for ease of site creation.
  • Integrate creative cloud management and dynamic media and assests handling.

Development Process

HERO Component.

Hero is a single component but has 11 different ways it can be styled into. The old CMS did not provide the flexibility to create these 11 layouts. Authors or site creators were restrcited to use the only given layout. I came up with a plan to make this component as flexible as possible and thus enrich the user experience.

Component creation WALKTHROUGH.

A theme based HERO component and a simple HERO component.

Steps
  • HANDOFF: The process involved getting designs and policy from Microsoft's design team.
  • ANALYSIS: Analyzing the requirements.
  • DEVELOP: Start the development work.
  • TEST: Communicate blockers to the design team and Adobe's engineering team.
How I planned this design.
Considerations:
  • As a site content creator, there should be flexibility to design pages and display products in different layouts.
  • In compliance to Microsoft and allowed layout styles there were a total of 11 ways to display the same information.
  • The aim was to make the design reusable to minimize the number of components created, avoid confusion of use and allow users to relate to a single model.
Process:
  • I created an AEM design that gives the user the option to choose a type of layout. E.g. Hero or Immersive Hero.
  • Based on the first selection the consecutive options change and thus you can design what elements can go onto your empty Hero canvas and drag and drop them around, resize them and change colors, text and fonts for all elements.
Advantages:
  • Usesr can now use a single canvas or layout and design the looks as needed.
  • Updates to a design do not require users to track which component was upated since it's only one.
  • Versioning updates are easy on a single design and require minimal engineering effort.
  • A single design helps maintain consistency along several pages and sites.
  • Testing is eaiser and faster for one design.
